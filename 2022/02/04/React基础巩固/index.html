Posted on 2022-02-04

React的好处

  1. 组件化
  2. 声明式编程
  3. 移动端
  4. 虚拟dom和diffing算法

原生js缺点

  1. 效率低
  2. 重绘重排
  3. 不好复用
  4. 命令式编程

用jquery的方式打开react

script中的type使用 text/babel
延伸：script有哪几种type？
jsx是语法糖 js语法扩展 js➕xml
虚拟dom是object react内部使用 转换为真实dom

严格模式 禁止自定义函数this指向window

类组件实例的三大属性

  • state
    类中自定义的方法默认开启局部严格模式，所以直接调用点结果为undefined
    类组件的方法是挂在原型上，需要将这些方法通过bind绑定在实例上

  • props
    给类增加propTypes来限制
    可以在类中通过static的方式在类上增加

  • refs

  1. 官方不推荐使用ref=字符串形式 性能问题
  2. 推荐使用回调函数形式
    内联绑定 每次更新调用两次 第一次为null 第二次有值
    在类中定义实例方法 不会出现null
  3. 使用createRef创建一个容器存储ref

非受控组件与受控组件

  1. 非受控组件 现用现取 使用ref
  2. 受控组件 随着输入维护状态 表单组件中通过onchange方法将值存入state中
    建议使用受控组件 减少使用ref

高阶函数

  • 传入的参数是函数或者返回的是函数
  • 常见高阶函数 promise settimeout arr函数等
  • 函数柯里化 函数调用继续返回的参数 多次接收 最后统一处理

新版react与旧版react的区别

  1. 即将废弃三个钩子，componentWillMount componentWillUpdate componentWillReceiveProps加上Unsafe前缀
  2. 新增了两个钩子
    getDerivedStateFromProps 需要加static 参数为state props 使用条件为state完全由props的值决定
    getSnapshotBeforeUpdate 在更新之前返回快照值 作为第三个参数传给 componentWillUpdate

Diff算法

React中key的作用？

作为标识 在更新时起作用
新旧虚拟Dom如何比较？

  1. 数据更新生成新的虚拟dom
  2. 新的虚拟dom通过key值与旧虚拟dom对比
  3. 如果key值一样且内容一样则复用
  4. 如果key值一样但内容不同则创建新的真实dom
  5. 如果key值不同也创建新的真实dom

使用index值作为key值存在的问题

  1. 效率不高，例如从数组头部（逆顺序添加）插入数据，那么能复用的就很少
  2. 如果结构中还包含其他例如input标签则会导致界面错误

如何选择key？🈶️id用id 如果只是列表渲染用index

react-ajax react只关注界面 不包含ajax请求

代理proxy解决跨域的原理

例如3000端口请求5000
请求可以正常发送 但在返回结果的时候被同源策略拦截
代理作为中间人不受同源策略的限制
相当于新起了一个3000端口的服务代替客户端请求和获取数据

React脚手架配置代理

  1. package.json中的proxy属性 但配置不了多个
  2. src文件夹下新建setupProxy.js配置具体规则target changeOrigin pathRewrite 能够配置多个代理
